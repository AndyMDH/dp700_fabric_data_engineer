# DP-700 Topic Summaries

One paragraph per topic area — use for a fast final review pass.

## Workspace configuration

Fabric workspaces sit on a capacity and hold items. Spark settings (pools, node family, autoscale, runtime version, high concurrency, environments) control notebook/job compute behavior. Domains group workspaces for tenant-level governance and discovery. OneLake workspace settings govern folder/file-level security (data access roles). Apache Airflow Job settings enable running managed Airflow DAGs inside Fabric for teams migrating from or needing native Airflow orchestration. Underneath all of this sits Fabric's 5-level hierarchy — Tenant → Capacity → Domain → Workspace → Item — with a matching 4-tier admin delegation model (Fabric admin, listed as "Power BI Administrator" in Microsoft Entra ID, → Capacity admin → Domain admin → Workspace admin) that lets a tenant-wide admin distribute day-to-day responsibility instead of managing every workspace directly.

## Lifecycle management

Git integration provides version control within a workspace (branch, commit, PR, merge), while deployment pipelines promote items across Dev/Test/Prod stage workspaces with optional deployment rules to swap environment-specific values automatically. Database projects bring schema-as-code to Warehouses via source-controlled `.sql` files. These three mechanisms are complementary, not competing — the recommended production pattern connects only the Development workspace to Git (avoiding sync conflicts across multiple stages) and uses "Branch out to new workspace" to give each developer an isolated branch *and* workspace at once, while deployment pipelines handle the Fabric-side promotion from Dev through Test to Production.

## Security and governance

Fabric evaluates access in three sequential levels: Microsoft Entra ID authentication → Fabric access → data security. Data security itself has four controls of increasingly narrow scope — workspace roles, item permissions, compute/granular permissions (where RLS, CLS, and dynamic masking live), and OneLake security. Critically, the first three all grant *all* data in an item once access is given; only OneLake security roles restrict access *within* an item, and every lakehouse ships with a built-in `DefaultReader` role that must be explicitly removed for a narrower custom role to actually take effect. RLS (T-SQL `CREATE SECURITY POLICY` + a filter-predicate function) applies to everyone by default, including admins, unless the predicate explicitly carves out an admin exception — and like CLS and dynamic masking (`default()`/`email()`/`partial()`/`random()` functions), it's vulnerable to side-channel inference unless layered with the other controls. A CLS-protected table also forces a Direct Lake semantic model to silently fall back to DirectQuery. Sensitivity labels (from Microsoft Purview) carry classification and can enforce protection, and propagate downstream to dependent items. Endorsement (Promoted vs Certified) signals quality/trustworthiness for discovery. All Fabric activity is captured in the Purview/M365 audit log.

## Orchestration

Choose Dataflow Gen2 for low-code/business-user transformation, notebooks for code-first large-scale transformation, and pipelines to orchestrate both plus copy/control-flow activities — on a schedule or an event-based trigger (e.g., file arrival). Parameters and dynamic expressions make a single pipeline/notebook reusable across many invocations; the ForEach-over-control-table pattern is the standard way to scale one templated pipeline across many source tables.

## Loading patterns

Full loads truncate-and-overwrite; incremental loads track a watermark or use Delta CDF to load only changes — incremental is the default choice once volume/frequency makes full loads too costly. Before loading, data typically passes through a staging layer that isolates cleaning/validation from production tables. Records are tracked with two kinds of keys: business keys (natural, source-system identifiers) and surrogate keys (system-generated integers that protect the warehouse from source-system key changes). Dimensional modeling targets a star schema with conformed dimensions and one of seven SCD types — Type 1 (overwrite) and Type 2 (new row per change, old row expired) are the two tested most, but the full set runs 0 (never changes) through 6 (Type 2 + Type 3 combined). Fact tables load after dimensions, since each fact row looks up dimension surrogate keys — and for a Type 2 dimension, it must match the version valid *at the time of the event*, not the current version. Streaming loads typically land via Eventstream into either a Lakehouse (for BI/Spark-style queries) or an Eventhouse (for high-velocity time-series queries).

## Batch ingestion and transformation

Store choice follows workload shape: Lakehouse for Spark/ML, Warehouse for transactional T-SQL, Eventhouse for streaming/time-series. A Fabric warehouse loads via three tools: data pipelines (Copy jobs), T-SQL (`COPY`, `CTAS`, `INSERT...SELECT`), or Dataflow Gen2. Every Lakehouse table is a Delta table — an open-source storage layer giving Spark relational semantics: full ACID transactions, CRUD operations, version history with time travel, and the ability to act as both a streaming source and sink (unlike most static relational tables). Transformation tool choice follows the same logic as orchestration (Dataflow Gen2 vs notebook vs T-SQL vs KQL), often organized around a medallion architecture — bronze (raw, unmodified, so you can always reprocess from source), silver (validated/deduplicated/integrated), gold (business-ready star schema) — where each layer serves a different audience and can be secured independently via OneLake data access roles or separate workspaces per layer. OneLake shortcuts reference data in place (zero-copy); mirroring continuously replicates external operational databases into OneLake Delta tables. Pipelines ingest primarily via the Copy Data activity. Data quality handling — deduplication, missing-data handling, late-arriving data — commonly converges on Delta's `MERGE INTO` as the upsert primitive.

## Streaming ingestion and transformation

Pick Eventstream for no/low-code ingestion with light transforms, Spark Structured Streaming for complex custom logic unified with batch code, and KQL update policies when the destination is already an Eventhouse. In Real-Time Intelligence, native tables give best performance at the cost of duplicating data; shortcuts avoid duplication at some latency cost, with query-accelerated shortcuts closing most of that gap via a maintained cache. `withWatermark` plus `window()` in PySpark implement tumbling, hopping/sliding, or session windows for time-based aggregation over late-arriving streaming data.

## Monitoring

The Monitoring Hub is the first stop for any item's run history — a live table of activity status (Succeeded/Failed/In progress/Cancelled), per-run details, and up to 30 days of historical runs for spotting new failure patterns or creeping duration increases; its Schedule failures page centrally configures email alerts (Contributor role minimum). For deeper analysis, workspace monitoring spins up a dedicated Eventhouse collecting raw diagnostic logs/metrics, queryable via KQL/SQL. The Capacity Metrics app is the first stop for capacity-wide slowness (CU smoothing/throttling). Each item type has its own deeper monitoring surface: Spark UI for notebooks, refresh history for Dataflow Gen2 and semantic models, replication status for mirroring, and KQL diagnostic commands for Eventhouse ingestion. Direct Lake semantic models need explicit fallback-to-DirectQuery monitoring since that failure mode doesn't show up as an outright error. Data Activator is the no-code alerting/automation mechanism across these signals — its rules combine a Monitor (attribute + summarization to smooth noisy readings), a Condition (threshold/change/range/missing-data, with an occurrence rule to catch only sustained problems), and an optional Property filter, and — unlike a plain email alert — can trigger an actual downstream Fabric activity as an automated response, not just a notification.

## Error resolution

Each item type has characteristic failure modes and a corresponding place to look: pipelines (activity-level errors in run history), Dataflow Gen2 (step-level Power Query errors), notebooks (Spark UI stage/task detail and skew diagnosis), Eventhouse (`.show ingestion failures`), Eventstream (per-node monitoring/data preview), T-SQL (query insights), and OneLake shortcuts (source connection/credential/permission checks). The general discipline is to isolate to the smallest failing unit and distinguish data-dependent failures from environment/config failures.

## Performance optimization

Lakehouse tables: OptimizeWrite (on by default) prevents the small-file problem at write time by writing fewer, larger files; `OPTIMIZE` (compaction) and `ZORDER BY` (data skipping via co-location) are after-the-fact maintenance; `VACUUM` cleans up stale files with a 168-hour default *and minimum* retention. V-Order is Fabric's write-time Parquet acceleration — **disabled by default in new workspaces** as of Microsoft's current docs (a training-module-sourced "on by default" claim is now stale), controllable at session/table/write-operation level in Spark or whole-database level in a Warehouse (irreversible once turned off there) — worth enabling deliberately for read-heavy/Direct Lake scenarios, not assumed as a default. Pipelines benefit from parallelism and avoiding unnecessary sequential/serial ForEach chains. Warehouses benefit from up-to-date statistics, sound distribution, `MERGE`-based upserts, and the right monitoring surface for the question you're asking — DMVs for what's running right now, query insights for what's been expensive over time. Eventhouses benefit from update policies, materialized views (which combine a precomputed part with a live delta so results stay both fast and current), and tuned caching/retention policies, plus KQL discipline (filter early on the biggest-eliminating condition first, project only needed columns, smaller table first in a join). Spark performance hinges on right-sized autoscale/executors, high concurrency mode, correct partitioning, broadcast joins for small-side joins, the Native Execution Engine, and skew mitigation (AQE). Query performance generally comes down to pushing filters/projections early and keeping Direct Lake models within their supported limits (including avoiding column-level security on Direct Lake tables) to avoid DirectQuery fallback.
